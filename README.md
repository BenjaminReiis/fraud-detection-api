🎙️ Podcast – Entrevista Técnica | PixFlow – Distributed Payment System
Host
Seja muito bem-vindo a mais um episódio. Hoje recebemos um engenheiro de software backend especializado em Java e arquitetura de sistemas distribuídos. Ele desenvolveu o PixFlow, um sistema de transferências financeiras inspirado na arquitetura do PIX, com foco em consistência financeira, alta disponibilidade e processamento seguro de transações.
Obrigado por aceitar o convite. Para começar, qual era o objetivo principal desse projeto?
Engenheiro Backend
Obrigado pelo convite.
O PixFlow nasceu para simular problemas que encontramos diariamente em bancos digitais e grandes instituições financeiras. O objetivo nunca foi apenas criar uma API para transferências, mas reproduzir os desafios técnicos de um ambiente de missão crítica.
Quando um banco processa milhares de transações por segundo, qualquer erro pode gerar inconsistência financeira, duplicidade de pagamentos ou até perda de dinheiro. O projeto foi pensado exatamente para resolver esses problemas utilizando arquiteturas modernas e boas práticas de Engenharia de Software.
Host
Muitos desenvolvedores constroem apenas um CRUD bancário. O que diferencia o PixFlow desses projetos?
Engenheiro Backend
A principal diferença é que o foco não está na funcionalidade de transferir dinheiro, mas na confiabilidade do processamento.
Em um CRUD tradicional basta atualizar duas contas.
Em um sistema financeiro real existem diversos cenários críticos:
duas transferências acontecendo exatamente ao mesmo tempo;
falhas de rede durante o processamento;
reenvio da mesma requisição pelo cliente;
indisponibilidade de serviços;
necessidade de rastrear absolutamente todas as operações.
O PixFlow foi desenvolvido justamente para tratar esses cenários.
Meu objetivo era aproximar o projeto do comportamento de uma plataforma financeira utilizada em produção.
Host
Qual foi exatamente o seu papel como engenheiro de software?
Engenheiro Backend
Atuei como responsável por toda a arquitetura backend.
Fiz o levantamento dos requisitos técnicos, modelei o banco de dados, defini a arquitetura distribuída, implementei as APIs, desenvolvi toda a lógica de negócio, mecanismos de autenticação, segurança, processamento assíncrono e estratégias de consistência financeira.
Também projetei a infraestrutura utilizando Docker, pensando desde o início em escalabilidade e facilidade de implantação.
Além do código, minha responsabilidade era garantir que todas as decisões arquiteturais estivessem alinhadas com requisitos não funcionais como disponibilidade, desempenho, segurança e tolerância a falhas.
Host
Vamos falar das tecnologias. Você utilizou Node.js e TypeScript. Sendo um especialista em Java, por que escolheu esse ecossistema?
Engenheiro Backend
Minha experiência principal é com Java, especialmente no desenvolvimento de sistemas corporativos.
Escolhi Node.js porque queria explorar outro ecossistema mantendo os mesmos princípios arquiteturais que aplico em projetos Java.
Os conceitos permanecem exatamente os mesmos.
Consistência transacional.
Controle de concorrência.
Arquitetura orientada a eventos.
Mensageria.
Escalabilidade.
Observabilidade.
O framework muda, mas os fundamentos da Engenharia de Software continuam iguais.
TypeScript trouxe tipagem estática, melhor organização do domínio e redução significativa de erros durante o desenvolvimento.
Foi uma oportunidade interessante para aplicar conceitos que normalmente utilizo em aplicações Java em um ambiente diferente.
Host
Quais foram os maiores desafios técnicos durante o desenvolvimento?
Engenheiro Backend
Sem dúvida, garantir consistência financeira.
Quando várias transferências ocorrem simultaneamente, existe o risco de duas operações modificarem o mesmo saldo ao mesmo tempo.
Isso pode gerar perda de dinheiro ou saldos incorretos.
Para evitar esse cenário implementei mecanismos de controle de concorrência, operações atômicas e validações de consistência durante toda a execução das transações.
Outro desafio importante foi impedir duplicidade de pagamentos.
Em sistemas distribuídos é comum o cliente reenviar uma requisição por causa de timeout.
Se o sistema simplesmente executar novamente a operação, o pagamento será realizado duas vezes.
Para resolver isso implementei idempotência utilizando chaves únicas de processamento.
Mesmo que uma requisição seja enviada diversas vezes, apenas uma transferência é efetivamente processada.
Host
Você também implementou processamento assíncrono. Qual era a motivação?
Engenheiro Backend
Nem toda operação precisa acontecer dentro da requisição principal.
Registrar histórico, publicar eventos, atualizar métricas ou enviar notificações podem ser executados posteriormente.
Isso reduz o tempo de resposta para o usuário e aumenta significativamente a escalabilidade.
Além disso, permite desacoplar responsabilidades e tornar a arquitetura muito mais resiliente.
Host
O projeto possui uma Dead Letter Queue. Para quem está ouvindo, qual a importância desse componente?
Engenheiro Backend
Em sistemas distribuídos nem toda mensagem será processada com sucesso.
Pode ocorrer indisponibilidade de banco, falha de rede ou erro inesperado.
Ao invés de perder essa mensagem, ela é enviada para uma Dead Letter Queue.
Isso permite analisar posteriormente o motivo da falha, corrigir o problema e reprocessar a operação com segurança.
Esse mecanismo aumenta bastante a confiabilidade da plataforma.
Host
Vamos falar sobre o PostgreSQL. Qual foi o papel dele dentro da arquitetura?
Engenheiro Backend
O PostgreSQL foi escolhido principalmente pelo excelente suporte às propriedades ACID.
Em sistemas financeiros não existe espaço para inconsistência.
Cada transferência precisa garantir atomicidade, consistência, isolamento e durabilidade.
Esses princípios são fundamentais quando estamos movimentando valores financeiros.
Host
E onde o Redis entra nessa arquitetura?
Engenheiro Backend
O Redis foi utilizado como uma camada de alta performance.
Ele auxilia no cache, controle de concorrência, bloqueios distribuídos e otimização de operações críticas.
Como trabalha em memória, possui tempos de resposta extremamente baixos, tornando-se um excelente complemento para aplicações financeiras que precisam responder rapidamente.
Host
Se esse projeto fosse evoluir para produção, quais seriam os próximos passos?
Engenheiro Backend
Eu faria a decomposição completa em microsserviços, adicionaria mensageria com Apache Kafka, implementaria observabilidade utilizando Prometheus e Grafana, distribuíria os serviços em Kubernetes e utilizaria uma estratégia de deploy contínuo com CI/CD.
Também adicionaria mecanismos de circuit breaker, service discovery, tracing distribuído e replicação geográfica do banco de dados para aumentar ainda mais a disponibilidade da plataforma.
Host
Para finalizar, qual foi o principal aprendizado que esse projeto trouxe para você?
Engenheiro Backend
O maior aprendizado foi perceber que desenvolver software para ambientes financeiros vai muito além de escrever código.
A verdadeira complexidade está em garantir que o sistema continue correto mesmo diante de concorrência, falhas de infraestrutura, perda de conexão, reprocessamentos e alto volume de transações.
Esse projeto reforçou a importância de tomar decisões arquiteturais baseadas em confiabilidade, escalabilidade e resiliência.
No fim, um sistema financeiro não é medido apenas pelas funcionalidades que entrega, mas pela capacidade de manter a integridade dos dados em qualquer cenário.

- 🛠️ Tratamento global de erros
- ❤️ Health check para monitoramento
- 🔄 Versionamento de API (/v1)
---

- Python
- FastAPI
- Uvicorn
- PyJWT
- Render (deploy em cloud)

}

authorization: seu_token_aqui

🚨 Real-Time Fraud Detection API
API de detecção de fraude em tempo real desenvolvida com foco em arquitetura de backend profissional.

PythonFastAPIJWTRender

🔗 Links
API Online: https://fraud-detection-api-jvwk.onrender.com
Documentação (Swagger): https://fraud-detection-api-jvwk.onrender.com/docs
Site do Projeto: https://benjaminreiis.github.io/fraud-detection-api/
💡 Sobre o Projeto
Este projeto simula como sistemas financeiros analisam transações bancárias para detectar possíveis fraudes em tempo real.

A aplicação foi construída utilizando boas práticas de desenvolvimento backend, incluindo autenticação segura com JWT, versionamento de API e tratamento global de erros — reproduzindo o comportamento de plataformas financeiras utilizadas em produção.

🚀 Funcionalidades
🔐 Autenticação com JWT
💳 Análise de transações em tempo real
🚨 Detecção de fraude baseada em regras
📊 Consulta de status por usuário
🧠 Lógica inspirada em Machine Learning
🛠️ Tratamento global de erros
❤️ Health check para monitoramento
🔄 Versionamento de API (/v1)
🧱 Tecnologias Utilizadas
Python — linguagem principal do projeto
FastAPI — framework web para construção da API REST
Uvicorn — servidor ASGI de alta performance
PyJWT — geração e validação de tokens JWT
Render — plataforma de deploy em cloud
🔐 Como Funciona a Autenticação
1. Faça login para obter o token
http 
POST /v1/auth/login
json 
{
  "username": "seu_usuario",
  "password": "sua_senha"
}
2. A API retorna um token JWT
json 
{
  "token": "seu_token_aqui"
}
3. Use o token nos endpoints protegidos
Envie o token via header em todas as requisições autenticadas:

authorization: seu_token_aqui
📡 Endpoints Principais
Autenticação
POST /v1/auth/login
Realiza o login e retorna um token JWT.

Transações
POST /v1/transactions
Submete uma transação para análise de fraude em tempo real.

GET /v1/transactions/status/{user_id}
Consulta o status das transações de um usuário específico.

Monitoramento
GET /health
Verifica a disponibilidade e saúde da aplicação.

🧠 Como Funciona a Detecção de Fraude
O sistema analisa cada transação com base em regras inspiradas em modelos de Machine Learning, avaliando fatores como:

Valor da transação — montantes incomuns ou fora do padrão do usuário
Frequência — múltiplas transações em curto intervalo de tempo
Localização — origem geográfica inconsistente com o histórico
Horário — transações em horários atípicos
Cada transação recebe uma classificação de risco e, quando identificada como suspeita, é marcada como potencial fraude.

🚀 Como Executar Localmente
Pré-requisitos
Python 3.10+
pip
Instalação
bash 
# Clone o repositório
git clone https://github.com/benjaminreiis/fraud-detection-api.git
cd fraud-detection-api

# Crie e ative um ambiente virtual
python -m venv venv
source venv/bin/activate   # Linux/Mac
venv\Scripts\activate      # Windows

# Instale as dependências
pip install -r requirements.txt
Executando a API
bash 
uvicorn main:app --reload
Acesse a API em http://localhost:8000 e a documentação em http://localhost:8000/docs.
